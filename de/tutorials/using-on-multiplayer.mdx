## Bereitstellungsmodell

- Installiere MineClawd auf dem Server, um `/mineclawd` und `/mclawd` zu aktivieren.
- Die Client-Installation ist für die Kernbefehle optional.
- Die Client-Installation ist erforderlich für:
  - GUI-Öffnungspaket von `/mineclawd config`
  - Buch-UI von `/mineclawd history`
  - Pop-up-Frage-UI für `ask-user-question`

<Info>
Wenn ein Spieler den Client-Mod nicht hat, bietet MineClawd trotzdem einen Fallback-Frageablauf im Chat (`/mineclawd choose ...`).
</Info>

<Info>
Ohne Client-Mod kannst du weiterhin `/mineclawd config <key> <value>` im Chat zur Konfiguration verwenden.
</Info>

## Berechtigungsmodell und Admin-Kontrolle

- MineClawd-Befehlswurzeln sind durch Vanilla-Berechtigungsstufe `2` geschützt (`source.hasPermissionLevel(2)`).
- Es gibt keine eingebauten benutzerdefinierten Berechtigungsknoten pro Unterbefehl.
- Praktisches Ergebnis: OP nur an vertrauenswürdige Operatoren vergeben.

Wenn dein Server-Stack Berechtigungs-Remapping auf Befehlsebene unterstützt, gib `/mineclawd` und `/mclawd` nur für deine AI-Operator-Rolle frei.

## Sichtbarkeitskontrollen für volle Server

MineClawd hat eine Sichtbarkeitseinstellung pro Spieler:

```bash
/mineclawd config broadcast-requests-to <self|all|ops>
```

- `self`: nur der Anfragende sieht Prompt-Echo und Aufgabenstatus.
- `all`: alle sehen es.
- `ops`: nur Operatoren sehen es.

Diese Einstellung wird in `mineclawd/player-settings.json` gespeichert und gilt pro Spieler-UUID.

Sie kann auch im GUI-Konfigurationsbildschirm im Abschnitt "Misc" als "Broadcast requests to" gesetzt werden.

## Session-Eigentum im Multiplayer

- Session-Daten sind pro Owner-Key isoliert.
- Spielerbefehle arbeiten mit dem eigenen Session-Set dieses Spielers.
- Konsolenbefehle nutzen einen separaten Owner-Namespace.

Für Team-Betrieb solltest du früh entscheiden, ob jeder Operator persönliche Sessions behält oder alle ein gemeinsames Operator-Konto teilen.

## Dynamic Registry auf dedizierten Servern

Standardmäßig ist `dynamic-registry-mode` auf `AUTO`. In `AUTO` ist die Runtime für dynamische Platzhalter auf dedizierten Servern deaktiviert.

Zum erzwungenen Aktivieren:

```bash
/mineclawd config dynamic-registry-mode enabled
```

Danach den Server neu starten.

### Was sich bei Aktivierung ändert

- Dynamische Platzhalter für Item-/Block-/Fluid-Slots werden zur Laufzeit verfügbar.
- Dynamischer State-Sync wird an Clients gesendet.
- Persistenter Weltzustand speichert dynamische Slot-Daten.
- Clients müssen ebenfalls MineClawd installieren, um diesem dedizierten Server beizutreten.

<Warning>
Aktiviere die Dynamic Registry auf Produktionsservern erst nach einem gestaffelten Test. Die Client-Pflicht ist eine harte Kompatibilitätsänderung für alle Spieler.
</Warning>

## Empfohlene Operator-Baseline

| Ziel | Empfohlene Einstellung |
| --- | --- |
| Chat-Rauschen reduzieren | `broadcast-requests-to ops` |
| Lange Tool-Loops verhindern | `limit-tool-calls true` |
| Loops begrenzen | `tool-call-limit 8` bis `12` |
| Nur Vorfälle debuggen | `debug-mode true` temporär |

## Rollout-Checkliste für Server-Betreiber

1. Lege zuerst deine Operator-Rolle und OP-Richtlinie fest.
2. Setze Provider- und Modell-Zugangsdaten.
3. Setze die Broadcast-Richtlinie (`self`, `ops` oder `all`).
4. Entscheide, ob Dynamic Registry auf `auto` bleibt oder auf `enabled` geändert wird.
5. Teste vor dem öffentlichen Start mit zwei echten Konten (Operator und Nicht-Operator).
