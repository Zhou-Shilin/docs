## Modelo de despliegue

- Instala MineClawd en el servidor para habilitar `/mineclawd` y `/mclawd`.
- La instalación en cliente es opcional para el uso básico de comandos.
- La instalación en cliente es obligatoria para:
  - paquete de apertura de GUI de `/mineclawd config`
  - interfaz de libro de `/mineclawd history`
  - interfaz emergente de preguntas para `ask-user-question`

<Info>
Si un jugador no tiene el mod de cliente, MineClawd aún tiene un flujo alternativo de preguntas en el chat (`/mineclawd choose ...`).
</Info>

<Info>
Sin mod de cliente, igualmente puedes usar `/mineclawd config <key> <value>` en el chat para configurar.
</Info>

## Modelo de permisos y control de administración

- Las raíces de comando de MineClawd están protegidas por el nivel de permiso vanilla `2` (`source.hasPermissionLevel(2)`).
- No hay nodos de permisos personalizados integrados por subcomando.
- Resultado práctico: da OP solo a operadores de confianza.

Si tu stack de servidor soporta reasignación de permisos por comando, expón `/mineclawd` y `/mclawd` solo al rol de operador de IA.

## Controles de visibilidad para servidores concurridos

MineClawd tiene un ajuste de visibilidad por jugador:

```bash
/mineclawd config broadcast-requests-to <self|all|ops>
```

- `self`: solo quien hace la solicitud ve el eco del prompt y el estado de la tarea.
- `all`: todos lo ven.
- `ops`: solo lo ven los operadores.

Este ajuste se guarda en `mineclawd/player-settings.json` y se delimita por UUID de jugador.

También puede configurarse en la pantalla de configuración GUI, en la sección "Misc", como "Broadcast requests to".

## Propiedad de sesiones en multijugador

- Los datos de sesión están aislados por clave de propietario.
- Los comandos de jugador operan sobre el conjunto de sesiones de ese jugador.
- Los comandos de consola usan un namespace de propietario separado.

Para operación en equipo, decide pronto si cada operador mantiene sesiones personales o si todos comparten una cuenta de operador.

## Registro dinámico en servidores dedicados

Por defecto, `dynamic-registry-mode` es `AUTO`. En `AUTO`, el runtime de marcadores dinámicos está deshabilitado en servidores dedicados.

Para habilitarlo de forma forzada:

```bash
/mineclawd config dynamic-registry-mode enabled
```

Luego reinicia el servidor.

### Qué cambia cuando se habilita

- Los marcadores dinámicos para slots de ítems/bloques/fluidos pasan a estar disponibles en runtime.
- La sincronización de estado dinámico se envía a los clientes.
- El estado persistente del mundo guarda datos de slots dinámicos.
- Los clientes también deben instalar MineClawd para entrar a este servidor dedicado.

<Warning>
Habilita el registro dinámico en servidores de producción solo después de una prueba por etapas. El requisito de cliente es un cambio de compatibilidad estricto para todos los jugadores.
</Warning>

## Base recomendada para operadores

| Objetivo | Ajuste sugerido |
| --- | --- |
| Reducir ruido en el chat | `broadcast-requests-to ops` |
| Evitar bucles largos de herramientas | `limit-tool-calls true` |
| Mantener bucles acotados | `tool-call-limit 8` a `12` |
| Depurar solo incidentes | `debug-mode true` temporalmente |

## Checklist de despliegue para propietarios de servidor

1. Define primero tu rol de operador y política de OP.
2. Configura credenciales de proveedor y modelo.
3. Configura la política de difusión (`self`, `ops` o `all`).
4. Decide si el registro dinámico se queda en `auto` o cambia a `enabled`.
5. Prueba con dos cuentas reales (operador y no operador) antes del lanzamiento público.
